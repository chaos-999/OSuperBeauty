## 一、如何自己读日志判分

### 1.1 评测脚本工作原理（30 秒版）

`judge_basic-glibc.py` 的核心逻辑：

```
逐行读串口输出
  ↓
看到 "========== START test_brk =========="  → 开始收集 test_brk 的数据行
  ↓
看到 "========== END test_brk =========="    → 停止收集，调用 test_brk.test(data)
  ↓
test 方法里依次执行 assert_equal / assert_in_str / assert_ge ...
每通过一个断言 → pass +1，失败 → 抛异常、后续断言不再执行
  ↓
最终输出 JSON: {"name": "test_brk", "all": 3, "pass": 3, "score": 3}
```

**关键点：** `all` = 这道题的总断言数，`pass`/`score` = 通过的断言数。**如果第一个断言就挂了，后面的不会跑。**

```

```

## 四、后续修改清单（按优先级排序）

### 🔴 P0：修复 RISC-V yield 串口输出交错（丢 4 分）

**现象：** 多进程并发写 UART 时行被截断拼接

**根因：** RISC-V 的 `consolewrite()` / UART 驱动没有加锁保护，或者 `printf` 没有做成原子操作

**修改方向：**
- `kernel/fs/console.c` 的 `consolewrite()` 里加自旋锁，整行输出原子化
- 或者 `kernel/util/printf.c` 里用缓冲区 + 一次性 write
- 参考 LoongArch 为什么没这个问题——对比两者的 console/uart 实现

**验证方式：** 重跑 Docker，检查 `yield` 的输出是否恰好 15 行、无拼接

### 🟡 P1：修复 execve 测试的 END 标记

**现象：** 输出 `========== END main ==========` 而非 `END test_execve`

**根因：** 测试二进制 `execve` 内部用的程序名是 `main`，printf 的格式字符串写死了名字

**修改方向：** 修改测试程序的源代码（在 `testsuits-for-oskernel` 仓库中），或者让 init 里 fork 出来的子进程在 exec 之前设置正确的程序名

**影响：** 当前不影响打分（脚本匹配 `"========== END " in line`），但为长期兼容应修复

### 🟡 P2：统一 mmap/munmap 的映射长度行为

**现象：** RISC-V file_len=27（精确），LoongArch file_len=4096（整页）

**修改方向：** 检查 `kernel/syscall/sysfile.c` 或 `kernel/mm/vm.c` 中 mmap 的实现，确认两种架构的页对齐逻辑是否一致。两种行为都能过断言，但不一致说明有架构相关 bug。

### 🟡 P3：统一 times 系统调用的时间统计

**现象：** RISC-V utime=12 stime=6，LoongArch utime=1 stime=0

**修改方向：** 检查时钟中断频率配置（`kernel/trap/rv/trap.c` vs `kernel/trap/la/trap.c`），确认 tick 递增逻辑一致

### 🔵 P4：LoongArch test_final 的 namei failed

**现象：** interrupts / copy-file-range / splice 测试全部 `namei failed`

**根因：** SD 卡镜像上可能没有这些测试二进制，或者路径不匹配

**修改方向：**
- 确认测试二进制是否在 `init-la.c` 指定的路径下（`/musl/` 和 `/glibc/`）
- 如果缺失，需要从 `testsuits-for-oskernel` 自行构建 SD 镜像
- 或者在 `init-la.c` 中调整 chdir 路径

### 🔵 P5：RISC-V 也补上 test_final

目前 RISC-V 的 `test_final()` 被 `#if 0` 包裹（因依赖 busybox），LoongArch 用的是直接 fork+exec 测试二进制的版本。应该给 RISC-V 也写一个类似的直接 exec 版本，不再依赖 busybox。

## 五、自助判分方法

以后跑完 Docker，拿到 `os_serial_out_rv.txt` 后，可以快速手工对照本节判断：

### 5.1 检查是否所有测试都跑了

```bash
grep "========== START" os_serial_out_rv.txt | wc -l
# 应该是 32（glibc）+ 32（musl）= 64
```

### 5.2 检查是否有断头测试（有 START 无 END）

```bash
grep "========== START" os_serial_out_rv.txt | wc -l
grep "========== END" os_serial_out_rv.txt | wc -l
# 两个数字应该相等
```

### 5.3 快速判定 yield 是否通过

```bash
# 抓取 START test_yield 到 END test_yield 之间的行数
sed -n '/START test_yield/,/END test_yield/p' os_serial_out_rv.txt | wc -l
# 应该是 17（含首尾标记行 + 15 行数据）
# 如果明显多于 17 → 有输出交错，yield 丢分
```

### 5.4 批量检查断言匹配情况

```bash
# 检查 chdir ret
grep -c "chdir ret: 0" os_serial_out_rv.txt   # 应该是 2（glibc + musc 各 1）

# 检查 fork
grep -c "parent process. wstatus:" os_serial_out_rv.txt  # 应该是 2

# 检查 pipe
grep -c "Write to pipe successfully" os_serial_out_rv.txt  # 应该是 2

# 全部 32 个测试的 START 标记
grep "========== START test_" os_serial_out_rv.txt
```
