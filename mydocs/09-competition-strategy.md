## 五、最优攻略路线（按投入产出比排序）

### 🔴 P0 — RISC-V yield UART 锁（投入 30 分钟，+8 分）

**为什么先修这个：** basic 满分是"正常人"的入场券。96% 和 100% 在排行榜上心理差异巨大——评委看到的是一支"基础扎实"的队伍 vs "还差一点"的队伍。

**怎么修：** `kernel/fs/console.c` 或 `kernel/drive/uart.c` 中给 `consolewrite` 加自旋锁，保证多进程不交错。

### 🔴 P1 — busybox sh applet（投入 1-3 天，+216 分，排名翻倍）

**这是整个比赛对你来说最重要的一个 bug。** 修好它：

- 一次解锁 **6 组测试**：busybox、lua、libctest、libcbench、lmbench、git
- 分数从 408 → 624，排名从 ~38% → ~25%
- **直接进入前 1/4，稳进复赛**

**怎么排查：**

```
1. 确认 SD 卡镜像上有 /glibc/busybox 且可执行
2. 手写一个最简单的 init 测试：execve("/glibc/busybox", {"busybox", "sh", "-c", "echo hello"}, env)
3. 如果 argv 传递有问题 → 修 kernel/proc/exec.c 的 argv 拷贝逻辑
4. 如果 busybox 报 "applet not found" → busybox 二进制可能缺 CONFIG_ASH，需要自行编译
```

### 🟡 P2 — lua + libctest（投入 0-2 天，+28~160 分）

busybox 修好之后，这两组大概率自动跑通——它们都是通过 `busybox sh xxx_testcode.sh` 启动的。如果不通，通常是某个 syscall 缺实现或返回不对。

### 🟠 P3 — libcbench + lmbench（投入 2-5 天，+100 分）

这两个是性能测试，不是功能测试。只要 syscall 能跑通就能拿基础分。优化是后面的事。

### 🔵 P4 — iozone（投入 3-7 天，+200 分）

文件系统读写性能。需要 ext4 实现稳定，不能有数据损坏。如果你的 ext4 是 lwext4 库，大概率能跑出基础分。

## 七、时间线建议

假设距初赛截止还有 N 天：

```
Day 1:    修 yield UART 锁 → basic 满分 → 提交 → 408 分
Day 2-4:  排查 busybox sh applet → 修复 → 提交 → 624 分（稳进复赛）
Day 5-6:  验证 lua + libctest → 提交 → 700+ 分
Day 7+:   挑 libcbench/lmbench/iozone 中有把握的 → 冲 900+
```

**不要浪费时间去碰 iperf/netperf/cyclictest。** 这些需要完整网络栈和实时调度，初赛阶段对 90% 的队伍都不现实。你的时间应该花在"投入 1 天、拿回 200 分"的 busybox 上，而不是"投入 2 周、拿回 5000 分但只有 5 支队伍能做到"的网络栈上。
